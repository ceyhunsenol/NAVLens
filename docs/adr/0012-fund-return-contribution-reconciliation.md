# ADR-0012: Fund Return Contribution Reconciliation

- Status: Proposed
- Date: 2026-07-23

## Context

In [ADR-0011](0011-aligned-portfolio-return-contribution.md), we established the canonical computation for `PortfolioReturnContribution` based on exact `ReturnPeriod` and component-level weights. The portfolio contribution gives us the observed portfolio return strictly derived from the published holdings and the covered security prices.

However, a fund also publishes its official daily unit price (`UnitPrice`), from which a "published fund return" over a `ReturnPeriod` can be calculated. The published fund return may differ from the aggregated `PortfolioReturnContribution`. The difference between the two is a critical analytical output that must be clearly defined and modeled.

This document defines the mathematical, architectural, and financial semantics of this difference (the "reconciliation residual").

## Current and Planned States

**Current:**
- `DecimalReturn` and `PortfolioReturnContribution` are present in `navlens-core`.
- `ReturnPeriod` and `PeriodDecimalReturn` are present in `navlens-calendar`.
- `ReturnContributionResult` is present in `navlens-application`.
- Core reconciliation arithmetic/result, application exact-period orchestration, and PyO3 bindings are implemented.

**Planned:**
- Python point-in-time provenance orchestration
- Formatter/CLI/dataset

## Decision

We propose the concept of **Reconciliation Residual** to represent the difference between the published fund return and the observed portfolio contribution for a specific exact period.

### Canonical Equation

```
reconciliation_residual = published_fund_return - observed_covered_contribution
```

### Financial Interpretation

The `reconciliation_residual` is **not**:
- A prediction error
- An Alpha (excess return against a benchmark)
- A tracking error
- A model failure
- Automatically classified purely as expenses/fees

Instead, this residual represents the aggregate unmodeled or unavailable economic realities of the fund, which may include any combination of:
- Returns generated by the **uncovered portfolio weight** (e.g., missing price series, unlisted instruments)
- Cash returns and yields
- FX effects on non-base currency assets
- Derivative instrument returns
- Fund expenses and management fees
- Taxes
- Corporate actions not fully reflected in the adjusted price series
- Valuation timing differences (e.g., market close vs. fund valuation time)
- Price staleness
- Holdings publication lag

### Exact-Period Invariants

Reconciliation can only occur if the published fund return and the observed covered contribution share the exact same `ReturnPeriod`. Any mismatch in periods is a fatal typed validation error.

### Type Safety

- **Published fund return**, **observed covered contribution**, and **reconciliation residual** are separate domains and must be distinct typed values.
- Naked tuple, dict, or float-only results are strictly forbidden.
- `UnitPrice` and `SecurityPriceObservation` cannot be used interchangeably.
- The Application result must carry the validated `ReturnPeriod` along with the canonical core result.

### Coverage Semantics

- **No Renormalization:** The observed covered contribution must **never** be scaled to 100% (e.g., by dividing by the coverage ratio).
- **No Zero-Fill:** The uncovered portion of the portfolio must **not** be assumed to have produced a 0.0 return. 
- **Full Coverage Interpretation:** Even when `return_coverage == 1.0` (full coverage), the residual may not be exactly zero. Valuation timing differences, expenses, and unmodeled internal fund dynamics can still produce a non-zero residual. A zero residual is not guaranteed and should not be forced. The residual must not be hidden to artificially boost confidence.

### Layer Ownership

- **navlens-core (Rust):** Owns the scalar financial subtraction, the canonical `PortfolioReturnReconciliation`-like result, published `DecimalReturn`, observed `PortfolioReturnContribution`, reconciliation residual, coverage preservation, and non-finite arithmetic validation. It has no concept of periods and does not depend on `navlens-calendar` types.
- **navlens-application (Rust):** Compares the periods of `PeriodDecimalReturn` and `ReturnContributionResult`. It owns the exact `ReturnPeriod` equality invariant. It produces a typed application error on a period mismatch. After validation, it delegates the scalar values to the core logic. The application result carries the validated `ReturnPeriod` alongside the core reconciliation result.
- **navlens-python (Rust/PyO3 crate):** Performs only thin bindings, type conversions, and error mapping. It does not select datasets or manage provenance.
- **Python package:** Owns dataset selection, publication-time safety, provider/source orchestration, and the provenance envelope. It must **never** perform financial subtraction.

### Point-in-Time Constraints

When computing the reconciliation historically, any data (holdings, security prices, or fund prices) published after the `prediction_timestamp` must not be used. Point-in-time constraints apply equally to the fund's published unit price as they do to the portfolio components.

### Determinism

Reconciliation must be pure and deterministic. The Rust core must not make implicit decisions regarding data provider selection or thresholding. The same input set must always yield the same reconciliation result.

### Error Semantics

- Mismatched periods are a strictly typed application error in `navlens-application`.
- Non-finite subtraction or core domain validation failures are core errors.
- Missing fund-price observations are reported at the source/dataset selection boundary in Python.
- The core does not produce a "missing fund price" error; it expects a ready, validated scalar input.
- Native/Python mapping errors must not be swallowed; they should propagate as proper Python exceptions.

## Consequences

- This decision plans to formally recognize the difference between "calculated holding returns" and "official fund returns" as a first-class modeling concept, avoiding the trap of calling it an "error."
- Renormalizing weights or zero-filling uncovered assets is strictly forbidden, ensuring the raw unadulterated reality is visible to the researcher.
- Financial subtraction is localized to Rust `navlens-core`, avoiding Python-side math fragmentation.

## Rejected Alternatives

- **Dividing covered contribution by the coverage ratio:** Rejected because it assumes the uncovered portfolio performs exactly identically to the covered portfolio, introducing severe bias.
- **Treating the uncovered portion as zero return:** Rejected because it incorrectly misrepresents the observed contribution by assuming specific performance for the uncovered weight.
- **Naming the residual "prediction error":** Rejected because we are calculating an observed historical reconciliation, not evaluating a predictive model's accuracy.
- **Assuming the residual is entirely an "expense":** Rejected because the residual includes unmodeled assets, timing differences, and cash returns, not just fees.
- **Writing the subtraction formula in Python:** Rejected to maintain the architectural boundary that financial domain logic and invariant enforcement belong in `navlens-core`.
- **Silently comparing mismatched periods:** Rejected. Period alignment must be strictly enforced.
- **Returning a naked float without provenance:** Rejected. We require a typed orchestration result that preserves alignment and coverage metadata.

## Test Strategy

- **Core:** Positive, negative, zero residual; coverage preservation; non-finite rejection.
- **Application:** Exact-period success and period mismatch.
- **PyO3:** Rust parity and typed error mapping.
- **Python:** Point-in-time provenance preservation.
- **Regression:** No-renormalization and no-zero-fill regressions.
